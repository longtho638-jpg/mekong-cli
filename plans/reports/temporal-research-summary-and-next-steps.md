## Key Findings

### 1. Temporal Architecture Overview

Temporal is a **durable execution platform** solving the distributed systems problem: *How to reliably execute long-running workflows that survive crashes, network failures, and infrastructure issues?*

**Core insight:** Temporal treats workflows like database transactions:
- **Deterministic workflows** = orchestration logic (no I/O)
- **Activities** = non-deterministic work (I/O, side effects)
- **Automatic checkpointing** = invisible state persistence via event history
- **Replay** = resume from arbitrary point without manual coordination

### 2. Critical Determinism Constraint

**Rule:** Workflows must be deterministic—given same input, always generate same command sequence.

**Why?** Temporal replays workflows from event history on recovery. Non-deterministic operations (RNG, API calls, timestamps) diverge during replay → errors.

**Consequences for mekong-cli:**
- ❌ LLM-generated plans are non-deterministic → can't be replayed
- ✅ Static YAML recipes are deterministic → safe to replay
- ✅ Wrap non-determinism in activities (e.g., UUID generation)

**Action:** Refactor `planner.py` to separate deterministic orchestration from LLM-based planning.

### 3. Python SDK Patterns (Most Relevant)

| Feature | Pattern | Value for CLI |
|---------|---------|--------|
| **Workflows** | Async class with `@workflow.defn` | Pure orchestration |
| **Activities** | `@activity.defn` functions | Non-deterministic work (shell, LLM) |
| **Checkpointing** | Automatic at activity boundaries | Resume from checkpoint on crash |
| **Retry policy** | Configurable exponential backoff | Resilience to transient failures |
| **Signals/Updates** | Runtime message handlers | Pause/resume long jobs |
| **Heartbeats** | Periodic progress updates | Long-running step visibility |
| **Child workflows** | Nested workflow execution | Modular recipe composition |
| **Continue-as-new** | Restart with next batch | Unbounded iteration |
| **Compensation** | Inline rollback logic | Saga pattern for distributed txns |

### 4. TypeScript SDK Differences

- **Function-based** (not class-based) workflow definitions
- **Rich interceptor system** for observability (4 types: inbound/outbound workflow/activity, client)
- **Pre-bundled workflows** (compile-time, not runtime)
- **OpenTelemetry integration** built-in

**For mekong-cli:** Python SDK is more suitable (CLI context = Python-first).

### 5. The Determinism Trap (Critical!)

**Real-world example from research:**
```python
# ❌ BREAKS ON REPLAY
@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order_id: str):
        # LLM call varies each execution
        steps = await llm_client.generate_steps(order_id)
        # Replay may get different steps → divergence error
        for step in steps:
            await workflow.execute_activity(execute_step, step)
```

**Fix:**
```python
# ✅ DETERMINISTIC
@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order_id: str):
        # Hardcoded orchestration
        steps = [
            ("validate", validate_order, order_id),
            ("payment", process_payment, order_id),
            ("shipping", create_shipment, order_id),
        ]
        for name, activity, *args in steps:
            await workflow.execute_activity(activity, *args)

# LLM planning moved to activity (non-deterministic OK)
@activity.defn
async def generate_order_steps(goal: str) -> list[str]:
    return await llm_client.generate_steps(goal)
```

**For mekong-cli:** This means recipes MUST be statically defined (YAML/JSON), not generated by LLM on each run. LLM can generate initial plan, but execution uses fixed recipe.

## Applying Temporal Patterns to Mekong CLI

### Current Architecture Issues

| Issue | Temporal Solution |
|-------|-------------------|
| Crash mid-plan → restart step 1 | Checkpointing (event history) |
| Transient failures fail immediately | Retry policy (exponential backoff) |
| Failed plan leaves artifacts | Compensation (saga pattern) |
| Sequential execution is slow | Parallel activities (asyncio.gather) |
| Long jobs appear stuck | Heartbeats (progress updates) |
| Can't pause/resume running plans | Signals (runtime message handlers) |
| Plans vary on re-execution | Determinism constraint (static recipes) |

### High-Impact Mapping

```
Temporal Concept             Mekong Implementation
─────────────────────────────────────────────────
Event History        ←→     execution_trace.json
Activity Execution   ←→     ExecutionStep (shell/LLM/API)
Workflow Execution   ←→     RecipeExecution
Retry Policy         ←→     Step.retry_policy (YAML)
Compensation         ←→     Step.compensation (Saga)
Heartbeat            ←→     Progress checkpoint (every 5s)
Child Workflow       ←→     execute_child_recipe()
Continue-as-new      ←→     Batch processing with restart
Signal               ←→     Signal file (pause/resume)
Side Effect          ←→     Wrap UUID/random in activity
```

## Implementation Strategy: 3-Phase Rollout

### Phase 1: Quick Wins (Week 1) — 9 Hours Total

**Goal:** Core reliability improvements.

**Tasks:**

1. **Execution Trace Checkpointing** (2 hours)
   - Add `ExecutionTraceStore` class
   - Persist JSON after each step
   - On restart, skip completed steps
   - **Impact:** Crash at step 5 of 10 → resume from step 6 (no wasted work)

2. **Exponential Backoff Retry** (3 hours)
   - Add `RetryPolicy` class
   - Configure per-step in recipe YAML
   - Implement backoff calculation + jitter
   - **Impact:** 80% reduction in manual retries for transient failures

3. **Compensation/Rollback** (4 hours)
   - Add `Step.compensation` field
   - Implement `CompensationExecutor` (Saga pattern)
   - Execute compensations in reverse on failure
   - **Impact:** Clean failure states, no manual cleanup

**Effort:** Can be done in parallel by 3 developers
**Risk:** Low (isolated modules, good test coverage)
**ROI:** Immediate—existing users benefit day 1

### Phase 2: Medium Patterns (Week 2) — 14 Hours

**Goal:** Performance + visibility improvements.

**Tasks:**

1. **Deterministic Plan Definition** (4 hours)
   - Separate orchestration from LLM planning
   - Static recipe YAML as source of truth
   - Optional LLM-generated recipes (stored as artifacts, not replayed)
   - **Impact:** Plans are reproducible + debuggable

2. **Activity Heartbeating** (6 hours)
   - Wrap long-running steps with periodic heartbeats
   - Checkpoint progress every 5 seconds
   - Resume with last checkpoint
   - **Impact:** Visibility into long jobs; fast recovery on crash

3. **Parallel Execution** (4 hours)
   - Support `parallel` step groups
   - Use `asyncio.gather()` to run independent steps concurrently
   - Aggregate results + failures
   - **Impact:** 50% faster execution for independent steps

**Effort:** 14 hours (can be split across 2 developers)
**Risk:** Medium (asyncio + subprocess coordination can be tricky)
**ROI:** High (major UX improvements)

### Phase 3: Advanced Patterns (Week 3+) — Optional

**Goal:** Next-level reliability + scalability.

**Tasks:**

1. **Continue-As-New** (8 hours)
   - Batch processing with automatic restart
   - Prevents unbounded history growth
   - **Impact:** Can run infinite loops (e.g., data processing)

2. **Message Passing (Signals)** (6 hours)
   - Pause/resume running recipes
   - Query status mid-execution
   - **Impact:** Interactive control of long-running jobs

3. **Interceptors** (4 hours)
   - Observability hooks (logging, tracing, metrics)
   - OpenTelemetry integration
   - **Impact:** Visibility into execution flow

**Effort:** 18 hours (low priority)
**Risk:** High (requires architecture changes)
**ROI:** Medium (nice-to-have, not critical)

## Decision Points

### 1. Should mekong-cli implement full Temporal server dependency?

**Answer: NO**

- **Cost:** Add Temporal server as infrastructure dependency
- **Complexity:** Deploy + manage separate service
- **Benefit:** Full workflow history visibility, distributed task queue

**Alternative (RECOMMENDED):** Implement local file-based checkpointing + JSON trace storage
- **Cost:** 20 hours dev effort
- **Complexity:** Pure Python, no external dependencies
- **Benefit:** Same fault tolerance, zero ops burden

### 2. Should recipes be LLM-generated or static YAML?

**Answer: Both, but different purposes**

- **Static YAML recipes** = Source of truth for execution (deterministic, replayable)
- **LLM-generated plans** = Artifacts for analysis (one-time generation, not replayed)

**Example workflow:**
```bash
# Generate plan from LLM (one-time)
mekong plan:hard "Deploy new feature" > plan.md

# Convert to static recipe
cat plan.md | mekong scaffold-recipe > recipe.yaml

# Execute recipe (deterministic, reproducible)
mekong run recipe.yaml
```

### 3. Should checkpointing be opt-in or automatic?

**Answer: Automatic (opt-out via flag)**

- Safer default (always recoverable)
- Minimal overhead (JSON append after each step)
- Users who want fresh runs use `--no-checkpoint` flag

### 4. What's the maximum retry attempts default?

**Answer: 5 attempts with 1s initial, 2.0x backoff, 60s max**

- Follows Temporal SDK defaults
- Handles most transient failures (network blips, quota limits)
- Configurable per-step in YAML

## Unresolved Questions

1. **Execution trace storage:** JSON file (simple, local) vs. SQLite (queryable)?
   - Recommendation: Start with JSON, migrate to SQLite if needed

2. **Recipe versioning:** How to handle recipe changes during execution?
   - Recommendation: Immutable recipe snapshot at start time; store in trace

3. **Multi-step compensation ordering:** What if compensation has dependencies?
   - Recommendation: For now, simple reverse order; add DAG support in Phase 3

4. **Heartbeat frequency tuning:** Is 5s optimal for all step types?
   - Recommendation: Make configurable; default 5s; allow per-step override

5. **Distributed execution:** Should CLI support remote workers (like Temporal)?
   - Recommendation: Out of scope for Phase 1-2; evaluate in Phase 3

6. **History pruning:** How long to retain execution traces?
   - Recommendation: Keep 30 days; older traces archived to S3

7. **Testing determinism:** How to verify a recipe is deterministic?
   - Recommendation: Add test mode that replays execution multiple times

4. **Compensation** implements distributed transactions (sagas)
5. **Parallelism** unlocks performance for independent steps
