# COO SOPs — Standard Operating Procedures

> Hướng dẫn vận hành cấp COO cho hệ thống algo-trader.
> Trung gian giữa CEO (chiến lược) và Trader (thực thi) — quản lý operations hàng ngày.

---

## Hierarchy — 4 Cấp SOPs

```
CEO SOPs     ← Tầm nhìn, portfolio, business model
    ↓
COO SOPs     ← THIS: Operations, monitoring, incidents, optimization
    ↓
Founder SOPs ← Budget, risk, strategy lifecycle
    ↓
Trader SOPs  ← Commands, execution, debug
```

**COO = Operations Officer. Đảm bảo hệ thống chạy trơn tru 24/7.**

## SOP-O02: System Health Monitoring

**5 pillars COO giám sát:**

```
1. EXCHANGE HEALTH
   ├── Connectivity: all exchanges reachable?
   ├── Latency: API response <500ms?
   ├── Rate limits: usage <65%?
   └── WebSocket: streams active?

2. BOT ENGINE HEALTH
   ├── TypeScript: tsc --noEmit = 0 errors?
   ├── Tests: pnpm test = all PASS?
   ├── Memory: process <500MB?
   └── CPU: process <50%?

3. STRATEGY HEALTH
   ├── Signal generation: active?
   ├── Strategy weights: balanced?
   ├── Alpha decay: Sharpe declining?
   └── Regime detection: accurate?

4. INFRASTRUCTURE HEALTH
   ├── Tôm Hùm daemon: running?
   ├── Proxy (9191): responding?
   ├── Disk space: >10% free?
   └── Network: stable?

5. DATA HEALTH
   ├── Price feeds: current?
   ├── Order books: depth sufficient?
   ├── Historical data: no gaps?
   └── Reports: generating correctly?
```

## SOP-O03: Incident Response Framework

### Severity Levels

| Level | Trigger | Response Time | Escalate To |
|-------|---------|--------------|-------------|
| **P0 Critical** | Trading halted, capital at risk | Immediate | CEO |
| **P1 High** | Exchange down, strategy failing | <15 min | Founder |
| **P2 Medium** | Performance degraded, warnings | <1 hour | Self-resolve |
| **P3 Low** | Minor issues, optimization needed | <24 hours | Self-resolve |

### P0 Playbook — Trading Halted
```
1. IDENTIFY: Which circuit breaker triggered?
   /trading:health
   /trading:debug "trading halted"

2. CONTAIN: Prevent further damage
   # Bot already auto-halted by circuit breaker
   # Verify no open positions need manual close

3. DIAGNOSE: Root cause
   /trading:coo:incident P0 "trading halted — circuit breaker"

4. RESOLVE: Fix and verify
   # Fix root cause
   # Paper trade 2h verify
   /trading:auto BTC/USDT paper

5. REPORT: Document incident
   # Auto-generated by /trading:coo:incident
```

### P1 Playbook — Exchange Down
```
1. /trading:health → identify which exchange
2. ExchangeRouter auto-failover → verify working
3. If failover failed → manual intervention
4. Monitor until exchange recovers
5. Log incident
```

### P2 Playbook — Performance Degraded
```
1. /trading:coo:perf → identify metric(s) declining
2. Check strategy weights → rebalance if needed
3. Check market regime → strategy may not fit
4. Adjust parameters or switch strategy
5. Monitor 24h
```

## SOP-O04: Performance Optimization

**Weekly performance tuning cycle:**

```
1. MEASURE
   ├── Execution speed: order placement latency
   ├── Fill rate: % orders filled at target price
   ├── Slippage: actual vs expected price
   ├── Signal accuracy: predicted vs actual movement
   └── Strategy alpha: excess return vs benchmark

2. ANALYZE
   ├── Which strategy produces best risk-adjusted returns?
   ├── Which exchange has best execution?
   ├── Which pairs have best spreads?
   ├── What time of day performs best?
   └── Are circuit breakers triggering too often/rarely?

3. OPTIMIZE
   ├── Adjust strategy weights (±0.05)
   ├── Adjust execution timing (peak hours)
   ├── Adjust exchange routing (best execution)
   ├── Adjust pair selection (highest alpha)
   └── Adjust circuit breaker thresholds

4. VERIFY
   ├── Backtest optimized config
   ├── Paper trade 24h
   ├── Compare before/after metrics
   └── Rollback if worse
```

## SOP-O05: Capacity Planning

**Khi nào cần scale infrastructure:**

| Signal | Threshold | Action |
|--------|-----------|--------|
| API rate usage | >60% sustained | Add exchange accounts / adjust phantom rate |
| Order volume | >100/hour | Review stealth parameters |
| Pairs monitored | >10 | Consider dedicated scan processes |
| Exchanges | >5 | Review connection pool config |
| Data storage | Reports >1GB | Archive old reports |
| Memory usage | >1GB process | Optimize data structures |

## SOP-O06: Change Management

**Trước KHI thay đổi hệ thống:**

```
1. PLAN
   ├── Mô tả thay đổi (config? code? strategy?)
   ├── Impact assessment (affected modules)
   ├── Rollback plan
   └── Testing plan

2. TEST
   ├── tsc --noEmit → 0 errors
   ├── pnpm test → all PASS
   ├── Backtest with new config
   └── Paper trade 2h

3. DEPLOY
   ├── Off-peak hours (avoid high-volume)
   ├── One change at a time
   ├── Monitor 1h after change
   └── Document in changelog

4. VERIFY
   ├── All health checks GREEN
   ├── Performance metrics stable/improved
   ├── No unexpected circuit breaker triggers
   └── Report to Founder/CEO if significant
```
